JoeScreen desktop runbook

What the macOS app does today, how to run it, and how to test & debug every feature. Verified against main @ a2f4264 · 2026-07-11.

JoeScreen is a CoScreen clone: a multiplayer screen-share tool where people join a session, share individual windows (or a whole display) into floating live windows on everyone's desktop, talk over voice, point with live cursors, draw annotations, sync clipboards, and — with consent — remote-control each other's shared windows. The macOS app is the full-featured client; iOS is a viewer that can also publish mic/camera/screen.

Just want to use it? Download the dmg. The released app is self-contained: it defaults to the production SFU (sfu.cheffing.dev), so you open it, click Join, and you're in a session — no toolchain, no local server, no bun run dev. That developer loop (building from source against a local SFU) is a separate path, covered in §2 only if you're hacking on the code.

What "current state" means. The macOS vertical slice is built and passing its gate (331 tests / 0 failures, 8 skipped offline). The dmg is a Release build pointed at the production SFU (https://sfu.cheffing.dev, which co-hosts the token server); a from-source Debug build points at a local SFU (ws://localhost:7880). This runbook is the desktop app; shipping the dmg (build/notarize/publish) lives in ship-runbook.html.

1Prerequisites & permissions

Toolchain

  • macOS 14.0+ (deployment target), Apple Silicon.
  • Xcode + command-line tools (xcodebuild), xcodegen, and bun (the monorepo task runner).
  • A local LiveKit server for dev: Homebrew livekit-server v1.13.3. bun run dev starts one automatically if none is running.

macOS privacy grants (TCC)

The app is non-sandboxed by design (Developer-ID, not App Store) so features are gated purely by TCC prompts at runtime — there are no capability entitlements to toggle. Grant these in System Settings → Privacy & Security:

PermissionNeeded forWhen it prompts
Screen RecordingSharing any window or displayFirst Share…; the app preflights with CGRequestScreenCaptureAccess() so the prompt is deterministic.
MicrophoneVoiceFirst mic-on.
CameraWebcam "bubble"First camera-on.
AccessibilityBeing remote-controlled (input injection)Only when you approve a control request. Injection needs the kTCCServicePostEvent grant — a human step, never automatic.

Ad-hoc re-signing resets grants (source builds only). Local Debug builds are ad-hoc signed, so a rebuild can present a different code identity and macOS may re-prompt for Screen Recording / Accessibility. If a share suddenly fails with a -3801 from ScreenCaptureKit, re-grant Screen Recording and relaunch. The signed dmg build keeps a stable identity, so it grants once.

2Running the app

There are two ways to run JoeScreen. Pick by what you're doing:

Path A — Download & run the dmg most people

  1. Download the dmg from the download page (cheffing.dev), open it, and drag JoeScreen to Applications.
  2. Launch it. Because it's a Release build, the Server URL is pre-filled with the production SFU (sfu.cheffing.dev) — you don't type anything.
  3. Click Join. You're in the demo room on the live SFU. Grant Screen Recording / Mic / Camera when a feature first asks.

That's it — no terminal, no bun, no local server. The app fetches its own token from the server and dials the SFU the server returns. Everything in §4 works from the dmg. To get a second participant, have someone else download the dmg and join the same room name on the same (default) server — or use the same session on a second Mac.

First-launch Gatekeeper. If the dmg build isn't yet notarized, macOS may block it on first open. Right-click the app → Open (or allow it in Privacy & Security). See ship-runbook.html for the notarization status.

Path B — Build & run from source developers only

This is for hacking on the code. It builds a Debug build that points at a local LiveKit SFU (ws://localhost:7880) with an in-process dev token — it does not touch the production server. Requires the toolchain: Xcode + xcodebuild, xcodegen, bun, and Homebrew livekit-server v1.13.3. Run from the repo root:

CommandWhat it does
bun run dev [room]The one-command dev loop: starts livekit-server --dev (reuses a running one), builds JoeScreen-macOS, and launches it joined to ws://localhost:7880 in room (default demo). Ctrl-C stops the SFU.
bun run app [room]Builds & launches another instance joined to the same local SFU — this is how you get a second participant on one machine. Each launch gets a fresh random identity so two instances don't collide.
bun run livekitStarts just the local SFU (no app).
bun run build / bun run testTurbo-driven build / test across the workspace.

The Swift project lives in apps/joescreen/; the Xcode project is generated from Apps/project.yml (the .xcodeproj is gitignored).

Manual build (when you need Xcode control)

# from apps/joescreen/
xcodegen generate --spec Apps/project.yml
xcodebuild -project Apps/JoeScreen.xcodeproj -scheme JoeScreen-macOS \
  -derivedDataPath build build
open build/Build/Products/Debug/JoeScreen.app

Two-participant testing on one Mac (source path). Run bun run dev (instance A), then bun run app in a second terminal (instance B). Both land in demo on the local SFU. Share a window from A → it appears as a floating window on B, and vice-versa. Fastest way to exercise nearly every feature. (With the dmg, you'd instead use two Macs, or one Mac + the iOS app, since Release builds can't dial a local server.)

3Joining a session

Four ways in, all resolving to the same DirectJoinParameters (server URL + room + a fresh identity + display name):

  • Join sheet — the default UI. Server URL, room, display name; identity is minted per join.
  • Launch args (automation/demo, source builds): open -n JoeScreen.app --args --join-url ws://localhost:7880 --room demo. Optional --share-window-id <id>, --share-display-id <id>, or --share-main-display auto-share right after join.
  • Deep link — joescreen://join?server=<url>&room=<name> (also accepts url=; handled via .onOpenURL). Identity is always fresh per joiner, so an invite link is safe to reuse. This is how Copy Invite Link shares a session with a dmg user.
  • Recents — the menu-bar and window menu list recent sessions; Copy Invite Link puts a joescreen://join?… URL on the clipboard.

Debug vs Release token flow. In Debug, the app mints a dev-key JWT in-process (DevTokenMinter) and dials the URL as-is — no token server. In Release, it fetches a token from the token server, which returns the authoritative SFU URL to dial (the token server and SFU can be different hosts). Release also rejects a non-TLS SFU scheme.

4Feature reference

Everything is reachable from the menu-bar extra (person.2.wave.2 icon), the Window menu, and the in-session control bar.

Window & display sharing

Share one window (ScreenCaptureKit picker) or a whole display. Each becomes a live floating window on every peer, bordered in the sharer's color. Hover to Share shares whatever window you hover.

Voice

Mic toggle + input-device picker in the control bar. Muted/unmuted state broadcasts to the roster.

Webcam bubbles

Camera toggle + device picker (incl. Continuity Camera / iPhone). Your self-preview is mirrored; peers see a camera tile.

Live cursors

Everyone's pointer is rendered on shared windows at ~60fps (coalesced, latest-wins), colored per participant.

Annotations (draw)

Toggle draw mode to ink over a shared window. Strokes are convergent per-author with undo/clear; late joiners get existing ink via the room snapshot.

Clipboard sync

Session-scoped, default OFF, never persisted. When on, copy on one machine → paste on another. Echo loops are suppressed.

Remote control

Request to drive a peer's shared window; the owner sees a consent prompt (Allow / Deny). Only on approval does input injection turn on. iOS can drive a Mac but can never be controlled.

Window management

Focus/reopen a shared window, Bring All to Front, Always on Top per window, Follow New Shares, and focus-all-shares-of a participant.

Roster & presence

Live participant list with per-person mic/camera state and colors; SharePlay group-session presence when available.

5Testing each feature

Baseline setup — you need two participants in one room. Either:

  • From the dmg: two Macs (or one Mac + the iOS app) each open JoeScreen, keep the default server, and join the same room (demo).
  • From source: bun run dev (A) + bun run app (B) on one Mac, both in demo on the local SFU.

Automated gate

# from apps/joescreen/ — the primary machine gate
swift build          # 6 library targets
swift test           # 331 tests (8 skip offline)

The pure logic (arbiter, coalescers, sync engines, wire codec, track naming) is unit-tested and framework-free. LiveKit integration tests are metadata-only and skip unless LIVEKIT_URL is set — real device capture hangs a headless host (no TCC prompt), so they never run camera/mic in CI:

livekit-server --dev &
LIVEKIT_URL=ws://localhost:7880 swift test --filter JoeScreenLiveKitTests

6Debugging & logs

Live logs

The app logs through AppLog on the unified logging subsystem com.joescreen.app. Stream it:

log stream --predicate 'subsystem == "com.joescreen.app"' --style compact

Look for lines from the share path (startSharing…, publishVideoTrack sink ready…, screenCaptureAccess=…) and the connection pump (state transitions, publish failures).

Common failure signatures

SymptomLikely causeFix

Reset commands

tccutil reset ScreenCapture     # re-prompt for Screen Recording
tccutil reset Accessibility     # re-prompt for input injection
tccutil reset Camera; tccutil reset Microphone

7Known gotchas

  • Non-sandboxed on purpose. Input injection needs kTCCServicePostEvent, which is sandbox-incompatible, so the Mac app is Developer-ID (not Mac App Store). No com.apple.security.device.* entitlements exist.
  • SharePlay/GroupActivities carries no media. It's used only for presence/signaling; all real-time media flows over the LiveKit SFU. If SharePlay presence is absent, media still works.
  • Info-macOS.plist is generated. XcodeGen emits it from project.yml (gitignored). Edit project.yml, never the emitted plist.
  • Mesh doesn't scale. All internet sessions go star-through-SFU; there is no mid-session topology migration. One encoder per shared window at the sharer, not per-peer.
  • iOS is a viewer. It publishes mic/camera/whole-screen and can drive a Mac, but can never be remote-controlled (no injection API on iOS).
