Installing & running JoeScreen

A native macOS + iOS shared-desktop app in the spirit of CoScreen — multiple people share individual app windows, each landing on every peer's desktop as a real, movable, live-video window.

Read this first. There is now a working macOS call app. Run livekit-server --dev, build the app with XcodeGen + Xcode, launch two instances, and one shares a window that the other renders live in a native window — with voice and cursors. The full media pipeline (capture → VP9 → SFU → decode → render) is verified against a live dev server; an iOS viewer app builds + runs. The remaining live gates need one-time human steps (Screen Recording / mic TCC grants, an iOS tap); SharePlay's runtime needs two devices on different iCloud accounts. Input injection, clipboard, draw, and terminal are the next phases.
Runs today Pending human / hardware

Git + a terminal

03 Build & test runs today

This is the real, verifiable gate: the shared logic package builds and passes its full test suite on any Mac — no hardware, no network, no server.

the machine gate
$ swift build     # 6 library targets; JoeScreenKit at Swift 6 + strict concurrency
$ swift test      # 117 tests, 5 skipped (need a server/TCC), 0 failures

Expected result:

expected output
Build complete!
Test Suite 'All tests' passed
     Executed 117 tests, with 5 tests skipped and 0 failures

These tests cover the load-bearing logic: the wire protocol and its six-channel matrix, per-sender input ordering & loss detection, the chunker for >15 KB payloads, cursor coalescing, session-coordination + bootstrap round-trips, coordinate mapping with the security clamp, owner-side input authorization (every deny path), bandwidth admission math, the codec-fallback state machine, clipboard diffing, terminal secret redaction, ICE batching, and signaling back-pressure. The 5 skips are the LiveKit integration + capture tests, which run green against livekit-server --dev (they skip rather than fail when it's absent). See TESTING.md for the full map.

04 Run the media server runs today

All real-time media (screen video, cursors, input, clipboard, terminal) travels over a self-hosted LiveKit SFU — never over SharePlay. For local testing you don't even need TLS certs.

Loopback (fastest — no certs, no domain)

local dev
$ livekit-server --dev   # in-memory keys, ws://localhost:7880

Full server (Docker)

infra/
$ cd infra
$ docker compose up   # pinned livekit-server v1.13.3 + JWT token endpoint

See infra/README.md for supplying a domain, TLS certs, and real API keys. Two things to know: the API secret lives only in the token server (never in the app binaries), and media is plaintext in the server's memory until end-to-end encryption lands in a later version — run it on infrastructure you control.

05 The apps runs today

The macOS call app builds and runs. Generate the Xcode project from Apps/project.yml (the .xcodeproj is generated, not committed), build, and launch two instances for a real call.

the demo — one Mac, two instances
$ brew install livekit && livekit-server --dev &
$ xcodegen generate --spec Apps/project.yml
$ xcodebuild -project Apps/JoeScreen.xcodeproj -scheme JoeScreen-macOS -derivedDataPath build build
$ APP=build/Build/Products/Debug/JoeScreen.app
$ open -n "$APP" --args --join-url ws://localhost:7880 --room demo --identity "$(uuidgen)"
$ open -n "$APP" --args --join-url ws://localhost:7880 --room demo --identity "$(uuidgen)"
# A: Share → pick a window (grant Screen Recording once).  B: watch it live.

The iOS viewer (viewer + voice only) builds for the simulator with xcodebuild -scheme JoeScreen-iOS -destination 'generic/platform=iOS Simulator' build.

The handful of things that still need a human or paired hardware:

  • One-time TCC grants. The sharer needs Screen Recording; voice needs the mic. These are system dialogs a person approves once on the dev Mac.
  • SharePlay runtime. The coordination layer compiles + is unit-tested against a fake, but a real session needs two devices on different iCloud accounts and the group-session entitlement — it cannot run in one place.
  • iOS can be a viewer, never a controller. There's no public API to inject input into other apps on iOS. That asymmetry is permanent and built into the design.

Every remaining item, plus the full F1–F14 acceptance and the ≤150 ms latency target, is a pending run-book in TESTING.md. Nothing there is marked done until it's actually observed.

06 How it fits together

One idea dictates the whole design: SharePlay can't stream media. So there are two planes.

Coordination plane — SharePlay

  • Start the session (also gives free FaceTime voice)
  • Presence & roster (opaque participant IDs)
  • Low-rate signaling: hand the media transport its connection info
▼ bootstrap { server URL, room, token } ▼

Media plane — self-hosted LiveKit SFU

  • Per-window video tracks (VP9 for text legibility, H.264 fallback)
  • Typed data channels: cursor · input · clipboard · terminal · draw
  • Every client dials out → NAT traversal solved by construction

The channel semantics are deliberately opposite per payload — cursor moves are unreliable/latest-wins, keystrokes are reliable/strictly-ordered — and that's enforced at the type level so a keystroke cannot be put on the lossy channel. Full detail in docs/architecture.md.

07 Where this is on the road

  1. Phase 0 — Foundation. API verification, green build + tests, architecture & decision docs, pinned deps, server infra.
  2. Phase 1 — Working calls. LiveKit transport (verified A→B video over a live SFU), ScreenCaptureKit capture, the macOS call app (share → live native window), voice, cursors, mirrored state, SharePlay coordination layer, iOS viewer. You are here.
  3. Phase 2 — Remote control, clipboard, security. Cross-machine input via the tested InputAuthorizer + coordinate clamp; byte-intact code paste.
  4. Phase 4 — Draw & shared terminal. Ink annotations; a PTY streamed as text (far lower latency than video).
  5. Phase 5 — iOS sharing & polish. Broadcast extension, fallback voice, rooms & invites.
  6. Phase 6 — Hardening at scale. Reconnection, the 3–10 person session at the bound, full hardware test pass.
