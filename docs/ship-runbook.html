1Web changes → cheffing.dev (the everyday path)

  1. Edit worker code under apps/cheffing-hub/ or apps/joescreen-download/.
  2. Sanity-check locally (optional): cd apps/<worker> && bun run dev, then bunx tsc --noEmit.
  3. Commit and push to main. That's it — CI does the rest.
  4. Watch it: gh run watch $(gh run list --workflow cloudflare-deploy-prod.yml --limit 1 --json databaseId -q '.[0].databaseId')
  5. Verify production: 
    curl https://cheffing.dev/version                 # {"env":"production"}
curl https://joescreen.cheffing.dev/version       # {"version":"…","env":"production"}

What triggers the deploy workflow

.github/workflows/cloudflare-deploy-prod.yml (“Production • Cloudflare”) runs on pushes to main touching:

  • apps/joescreen-download/** or apps/cheffing-hub/** → deploys only the changed worker(s)
  • bun.lock or the workflow file itself → deploys all workers
  • Manual recovery: gh workflow run cloudflare-deploy-prod.yml (or the Actions “Run workflow” button) → deploys all workers. Use this after a failed run or a secrets rotation — no dummy commit needed.

First pushes and force-pushes (no usable diff base) also deploy everything rather than silently skipping.

Manual deploy fallback (bypasses CI)

# creds: .env.production at repo root (CLOUDFLARE_API_TOKEN + CLOUDFLARE_ACCOUNT_ID)
set -a; source .env.production; set +a
cd apps/cheffing-hub        && bun run deploy    # cheffing.dev + www
cd apps/joescreen-download  && bun run deploy    # joescreen.cheffing.dev

Always bun run deploy, never bare wrangler deploy. Both wrangler configs use the same worker name at the top level and under env.production, so a bare wrangler deploy (no --env production) overwrites the production worker with development vars.

Rules that keep CI green (learned the hard way)

  • One lockfile. The workers are bun workspace members; the root bun.lock is the single source of truth and CI installs once from the repo root with --frozen-lockfile. Never create per-app bun.lock files — that split is exactly what broke the first two deploy runs.
  • Adding a dependency: cd apps/<worker> && bun add <pkg>, then commit the changed package.json and the root bun.lock. Check with bun install --frozen-lockfile at the root before pushing.
  • Keep the directory lowercase apps/. Git tracks lowercase; the old on-disk Apps/ spelling hid the workers from bun's workspace glob on macOS while CI (case-sensitive) saw them — which is how the lockfile drift went unnoticed. (The inner apps/joescreen/Apps/ Swift directory is capital-A on purpose — different thing.)
  • Secrets rotation: update .env.production, run bun run secrets:push (pushes CLOUDFLARE_API_TOKEN + CLOUDFLARE_ACCOUNT_ID as environment secrets into the GitHub production environment), then gh workflow run cloudflare-deploy-prod.yml.

2New macOS build → downloadable at joescreen.cheffing.dev

One manual step blocks this today: enable R2 on the Cloudflare account. The API returns code 10042 — “Please enable R2 through the Cloudflare Dashboard”. Dashboard → R2 → purchase the free plan (needs a billing profile; free tier is genuinely free). Until then the production worker deploys without the DMG binding and /download returns 404 with a friendly message. A notarized JoeScreen.dmg (20 MB, built Jul 8) is already waiting in apps/joescreen/build/dist-mac/.

After enabling R2, in order:

bunx wrangler r2 bucket create joescreen-downloads → restore the commented-out r2_buckets block under env.production in apps/joescreen-download/wrangler.jsonc → push (CI redeploys) → bun run publish:dmg.

  1. bun run dist:mac — archives with the Developer ID Application cert from the login keychain, packages the dmg, notarizes it (notarytool --wait, same ASC key as TestFlight), staples. Interactive: expect one keychain password prompt — click “Always Allow”.
  2. bun run publish:dmg — uploads to r2://joescreen-downloads/JoeScreen.dmg. The live site serves the new binary immediately; clients with a cached copy get it within 5 minutes (max-age=300).
  3. Bump APP_VERSION in apps/joescreen-download/wrangler.jsonc — in both places (top-level vars and env.production.vars) — and push to main so the page shows the new version.
  4. Verify: curl -I https://joescreen.cheffing.dev/download (expect 200 + content-disposition: attachment) and curl https://joescreen.cheffing.dev/version.

Steps 2 and 3 are independent: the R2 upload changes what users download; the worker deploy only updates the displayed version.

3iOS changes → TestFlight

All creds come from .env.testflight at the repo root (APPLE_TEAM_ID, ASC_API_KEY_ID, ASC_API_ISSUER_ID, ASC_API_KEY_PATH → the .p8). Bundle IDs are fixed: iOS com.joescreen.app.ios, macOS com.joescreen.app. ASC app record: “JoeScreen”, id 6788803912. First build to pass Apple processing: build 7 (2026-07-08). Public tester link: testflight.apple.com/join/C3AdpdZJ.

Steady state (already set up — this is the whole loop)

  1. bun run testflight ios — bumps the build number in apps/joescreen/Apps/project.yml, archives, exports the .ipa, uploads via altool.
  2. Commit the bumped project.yml (currently at build 9). TestFlight rejects reused build numbers; an uncommitted bump means the next machine/checkout reuses one and the upload fails.
  3. Wait ~5–15 min for App Store Connect processing, then bun run testflight:review ios — attaches the build to tester groups and (unless configured off) submits the one-time external Beta App Review (~1 day).
  4. Upload failed but the build succeeded? Retry just the upload: bun run testflight:submit ios (reuses apps/joescreen/build/last-artifact-ios.txt).

Both platforms in one command

bun run release          # macOS dmg first (interactive keychain), then iOS → TestFlight, headless
bun run release --no-review   # skips the entire tester-groups + review step, not just the review

release.sh does not publish the dmg — bun run publish:dmg and the APP_VERSION bump (section 2, steps 2–3) are separate follow-ups. Also: the review step inside release.sh is wrapped in || true, so “iOS ✓ uploaded” in its summary does not prove the review step worked — check that step's own output.

First-time setup on a new machine / account (already done here)

bun run ship:setup ios — provisions bundle IDs and app records idempotently, then builds and uploads. --dry-run previews (and skips the app-record step too, not just build+upload).

Apple limits the scripts can't automate

  • App records: ASC API keys can't POST /v1/apps (403) — the setup script falls back to fastlane produce, which needs APPLE_ID + interactive 2FA once.
  • SharePlay (com.apple.developer.group-session) is a managed capability needing manual Apple approval — no API. Until approved, ships use minimal entitlements (viewer + voice, no SharePlay); afterwards pass SHIP_ENTITLEMENTS=full.
  • App Groups have no ASC API — fastlane produce group -g group.com.joescreen.app -n "JoeScreen App Group" or the portal. Only needed for the future ReplayKit broadcast extension; does not block TestFlight.
  • altool only reads keys from fixed dirs — the submit script stages the .p8 to ~/.appstoreconnect/private_keys/AuthKey_<KEYID>.p8, and only if that file is absent: it never refreshes an existing copy, so if a staged key ever goes bad, delete it there and re-run.
  • Testers need a reachable SFU: a TestFlight build pointed at localhost can't join calls. Deploy the apps/livekit docker-compose stack behind a public domain + TLS, or use bun run tunnel:testflight for an ephemeral smoke test from this Mac.

When something breaks

SymptomCause & fix
error: lockfile had changes, but lockfile is frozen in CI Root bun.lock is out of sync with a workspace package.json. Run bun install at the repo root, commit bun.lock. If a per-app bun.lock has reappeared, delete it.
Deploy step fails with auth errors Secrets missing/rotated in the GitHub production environment → bun run secrets:push, then re-run via gh workflow run cloudflare-deploy-prod.yml.
wrangler: custom domain “already attached to another script” Another worker owns the hostname. List: GET /accounts/$ID/workers/domains; detach: DELETE /accounts/$ID/workers/domains/<domain-id> (both with Authorization: Bearer $CLOUDFLARE_API_TOKEN), then redeploy.
New subdomain doesn't resolve locally right after deploy Negative DNS cache. Check reality with dig @1.1.1.1 <host>; your resolver catches up within minutes.
R2 commands fail with code: 10042 R2 not enabled on the account — dashboard step, see section 2.
altool: “Cannot determine the Apple ID from Bundle ID” No ASC app record for that bundle ID yet. bun run ship:setup ios (or create in ASC → Apps → New App), wait ~2 min, re-run bun run testflight:submit ios.
TestFlight upload rejected: build number already used Someone forgot to commit the bumped apps/joescreen/Apps/project.yml. Re-run bun run testflight ios (it bumps again) and commit this time.
Upload succeeded but the build turns INVALID in ASC (rejection email) Apple's post-upload static analysis — it never fails locally, so always check processing state, not just the upload. Past hits: missing NSCameraUsageDescription/mic strings (LiveKit references the APIs even though the viewer never films — ITMS-90683), icon with alpha, missing iPad orientations, and the macOS sandbox rejection (ITMS-90296, see stale-docs note).
Local hub links send you to production Fixed 2026-07-10 (devHref in apps/cheffing-hub/src/page.ts). If it recurs for a new tool card, give the card a devHref.

6What changed on 2026-07-10 (why deploys work now)

  • Root cause of the failed deploys: the on-disk Apps/ capitalization hid the workers from bun's apps/* workspace glob on macOS, so the root bun.lock never learned about them, while the case-sensitive CI runner treated them as workspace members — bun install --frozen-lockfile died on every run. Fixed in commit b39c325: directory normalized to apps/, per-app lockfiles removed, root lockfile regenerated, CI installs once from the root.
  • Deploy workflow gained workflow_dispatch (manual deploy-all), plus deploy-all on bun.lock changes and on pushes with no usable diff base.
  • GitHub production environment secrets pushed (they had never been set — the second latent failure).
  • cheffing.dev + www custom domains detached from the old www-production Next.js worker and attached to cheffing-hub. Old site preserved at preview.cheffing.dev.
  • joescreen-download deployed for the first time (DNS for joescreen.cheffing.dev created); its R2 binding is commented out under env.production until R2 is enabled, with /download 404ing gracefully.
  • Hub links became environment-aware — local dev no longer redirects to production.
